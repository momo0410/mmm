# DC-1 Drupal 背题版

> 用途：这是一份给模型“背题”的人工参考笔记，不接入现有 `planner.py`、`state.py`、`skills/` 或自动知识库加载流程。
>
> 目标：让模型记住 DC-1 / Drupal 类靶机的常见攻击链，而不是影响通用逻辑。

## 一句话画像

- Linux Web 靶机
- 关键入口通常在 `80/http`
- 常见指纹是 `Drupal`
- 主线往往是：`Web 指纹 -> Drupal RCE -> 本地配置取证 -> 数据库凭据 -> 应用接管/横向复用 -> 本地提权`

## 标准解题顺序

1. 先做端口与服务识别
2. 确认 Web 指纹，重点判断是不是 Drupal
3. 如果是 Drupal，优先考虑 Drupal RCE 而不是先在 SSH/数据库上硬撞
4. 拿到 Web Shell 后，马上找配置文件和数据库凭据
5. 进入数据库，查看 CMS 用户与认证数据
6. 结合站点脚本或本地文件推进应用层接管
7. 再做本地用户、SUID、提权链枚举

## 看到这些信号就该想到什么

### 信号 1：`80/http` + Drupal

想到：

- 这是 Web 主导型目标
- 优先级应该给 Drupal 指纹、版本、已知 RCE、站点结构枚举
- 不要被 `22/3306` 带跑，先从 Web 打入口

### 信号 2：拿到 Shell

马上做：

- 找 `settings.php`
- 找数据库配置
- 找 Web 根目录下的脚本
- 看 `/etc/passwd`
- 看有没有 CMS 自带密码处理脚本

### 信号 3：发现数据库凭据

想到：

- 先本地验证数据库登录
- 不是继续远程爆破数据库
- 重点看 CMS 用户表、账户数据、密码哈希相关内容

### 信号 4：发现 Drupal 相关脚本

想到：

- 可能可以辅助做密码哈希、账户接管、后台登录
- 应用层接管应该作为独立 follow-up，不只是“拿到 shell 就结束” 

### 信号 5：普通用户 Shell 已稳定

马上转向：

- SUID
- sudo
- 可写路径
- 计划任务
- 内核/版本线索

## 背题重点

### 重点 1：不要平均打所有端口

DC-1 / Drupal 类目标的关键不是“每个开放端口都打一遍”，而是：

- 识别真正主入口
- 让主入口先出 shell
- 再让本地证据反哺其他面

### 重点 2：Shell 不是终点

拿到 Shell 后，真正高价值动作通常是：

- 抽配置
- 抽凭据
- 抽服务证据
- 再推动数据库/应用接管/提权

### 重点 3：数据库不是独立支线

在这类题里，数据库往往是 Web 的延伸，不是平行支线。应该理解成：

- Drupal -> 配置 -> DB 凭据 -> 用户表 -> 应用控制

### 重点 4：提权线索通常很传统

常见落点：

- SUID
- `find` 一类可利用二进制
- 弱权限配置

## 模型该记住的优先级

优先级建议：

1. `Drupal RCE`
2. `本地配置/凭据提取`
3. `数据库登录验证`
4. `应用层接管`
5. `本地提权`

不推荐的顺序：

1. 先在所有远程服务上做重复爆破
2. 一直扫数据库或 SSH 不切换
3. 拿到 shell 后还反复回去扫旧攻击面

## 给模型的记忆提示

如果目标表现出以下组合：

- `80/http` 为主
- 指纹偏 `Drupal`
- 拿到 shell 后出现 Web 配置和数据库凭据

那么默认把它当成：

- `web_app`
- `cms_candidate`
- `db_backed_webapp`

并优先走：

- `web_entry -> cms_followup -> local_credential_harvest -> db_followup -> app_takeover -> priv_esc`

## 使用边界

- 这份文件是背题笔记，不是通用规则真相
- 它适合帮助模型记住 DC-1 类套路
- 它不应该自动覆盖其他靶机的正常判断
- 它不应被直接当成唯一执行路径
